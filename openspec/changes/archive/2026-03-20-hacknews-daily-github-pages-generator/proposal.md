## Why

当前仓库几乎为空，尚未形成明确的 OpenSpec 需求边界，而目标产品方向已经通过前期探索收敛：不是做一个普通的 Hacker News 列表页，而是做一份可每日阅读、可长期归档的 HN Daily Digest。

提出本变更的原因是：

- 需要把“信息抓取”明确升级为“日报编排”，让产出从原始链接集合变成有阅读优先级的日报页面。
- 需要定义一个适合 GitHub Pages 发布与归档的内容形态，使首页即今日日报，同时保留历史日报可回溯。
- 需要为后续实现建立统一的产品目标，避免把范围扩散到通用资讯站、实时流式榜单或多语言重写系统。
- 需要在 MVP 阶段优先验证“日报价值”而不是“页面复杂度”，聚焦每日精选、分栏组织和稳定归档。
- 需要明确内容风格边界：标题保持原始英文标题，不生成中文标题；仅在部分重点栏目中可选增加一句中文摘要或推荐理由，以增强阅读引导但不过度加工原内容。

## What Changes

本 proposal 将 OpenSpec 的目标定义为一个 “Hacker News 资讯日报 GitHub Pages 生成系统”，其 MVP 输出为每日一份静态日报，而不是持续更新的普通聚合列表页。

本次变更将明确以下产品范围：

- 每天自动从 Hacker News 拉取候选 stories。
- 对候选内容进行标准化、过滤与分栏整理，形成一份结构化日报。
- 首页直接展示“今日日报”。
- 历史日报归档并发布到 GitHub Pages，形成可连续访问的日报集合。
- 日报内容至少包含以下栏目：
  - 今日头条
  - 热门讨论
  - 值得深读
  - Ask HN
  - Show HN
- 页面标题展示原始标题，不生成中文标题。
- 可选仅在“今日头条”和“值得深读”栏目中，为条目补充一句中文摘要或推荐理由。

同时，本次变更也隐含排除了以下非目标方向：

- 不将 MVP 定义为普通的 Hacker News 列表页或榜单镜像页。
- 不要求对所有条目生成完整中文改写。
- 不在 proposal 中扩展到复杂个性化推荐、实时刷新或重度交互能力。

## Capabilities

### New Capabilities

- `daily-digest-generation`: 系统能够按天生成一份面向阅读的 Hacker News 日报，而不是仅输出原始条目列表。
- `story-normalization-and-curation`: 系统能够对拉取到的候选 stories 进行标准化、过滤与筛选，产出适合日报消费的内容集合。
- `digest-section-classification`: 系统能够将日报内容组织为固定栏目，包括今日头条、热门讨论、值得深读、Ask HN 与 Show HN。
- `homepage-as-latest-digest`: 系统能够将 GitHub Pages 首页定义为最新一期日报的入口，突出“当天可读”的核心体验。
- `digest-archive-publishing`: 系统能够发布并保留历史日报归档，使用户可以按日期回看过往内容。
- `selective-chinese-summary-support`: 系统能够在今日头条和值得深读栏目中，选择性附加一句中文摘要或推荐理由，同时保持原始标题不变。

### Modified Capabilities

- 无

## Impact

本变更将把项目的产品定位明确为“日报生成系统”而非“资讯列表站”，并为后续 specs 与实现工作提供清晰边界。

预期影响包括：

- 产品目标更聚焦，后续讨论将围绕“日报质量与可读性”展开。
- MVP 范围更清晰，有利于优先验证每日编排、栏目结构与归档价值。
- 内容规范得到统一，避免后续在标题翻译、摘要范围和页面形态上反复摇摆。
- GitHub Pages 的发布形态得到提前约束，便于后续围绕首页、日报页和归档页建立一致规格。
- 因当前仓库缺少既有 specs，本 proposal 将作为该方向的首个正式需求基线。
