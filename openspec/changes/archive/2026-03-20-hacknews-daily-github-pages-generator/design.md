## Context

OpenSpec 变更 `hacknews-daily-github-pages-generator` 的目标，是定义一个面向 GitHub Pages 的 Hacker News Daily Digest 生成系统。系统以“每日一份静态日报”为核心产物，通过定时任务自动拉取 Hacker News 候选内容，经过标准化、筛选、栏目编排后，输出今日首页与历史归档页面。

该系统的边界清晰：

- 输入来源是 Hacker News 当日候选 stories 与其可用元数据。
- 输出目标是可直接托管在 GitHub Pages 上的静态页面。
- 运行方式以离线批处理为主，而非在线实时服务。
- 发布方式依赖 GitHub Actions 的定时构建与静态产物部署。
- 内容呈现以英文原始标题为主，不做中文标题翻译。
- 中文内容仅作为可选增强，且只允许出现在“今日头条”和“值得深读”栏目中，以一句中文摘要或推荐理由的形式存在。

本设计重点不在爬虫实现细节，而在于定义稳定的技术方向、内容模型、页面组织与失败降级策略，以支撑后续 capability specs 与实现任务拆分。

## Goals / Non-Goals

**Goals:**

- 每天自动生成一份 Hacker News 日报。
- 首页始终展示最新一期日报内容。
- 历史日报可按日期持续归档与访问。
- 使用统一内容模型承载采集、标准化、筛选与栏目编排结果。
- 固定输出以下栏目：今日头条、热门讨论、值得深读、Ask HN、Show HN。
- 标题只展示原始标题，不生成中文标题。
- 仅在“今日头条”和“值得深读”中支持可选的一句中文摘要/推荐理由。
- 系统适合运行在 GitHub Actions + GitHub Pages 的低运维静态发布模式中。
- 当外部数据不完整或构建部分失败时，系统应尽量降级生成，而不是整站不可用。

**Non-Goals:**

- 不构建在线动态 Web 服务。
- 不提供实时流式刷新或分钟级更新。
- 不做用户登录、个性化推荐、收藏或评论交互。
- 不做完整的多语言国际化。
- 不要求对所有 Hacker News 内容做全量镜像或长期缓存。
- 不要求自动生成中文标题。
- 不要求在 MVP 中引入复杂机器学习排序或人工后台运营系统。
- 不要求替代 Hacker News 原站的讨论体验；日报仅做聚合与导航。
- 不要求在 MVP 中提供编辑器 UI；可选中文摘要可由数据输入或配置方式提供。

## Decisions

### 1. 采用离线生成，而不是在线请求时动态渲染

系统以“每日构建一次静态产物”为主。原因：

- Hacker News 日报天然是批处理问题，不需要请求时计算。
- 静态页面更适合 GitHub Pages 托管。
- 构建结果稳定，可回溯、可归档、易调试。
- 可以把失败范围限制在单次构建，而不影响历史已发布内容。

### 2. 使用 GitHub Actions 定时构建作为自动化调度层

系统通过 GitHub Actions 触发每日构建，负责：拉取候选 stories、执行标准化与筛选、完成栏目编排、生成静态页面、发布到 GitHub Pages。

原因：

- 与 GitHub Pages 的部署链路天然兼容。
- 不需要单独维护服务器或 cron 基础设施。
- 构建日志、失败记录、重试历史都可在仓库内审计。
- 成本低、操作面小，符合 MVP。

### 3. 采用统一内容模型贯穿全流程

系统需要一个统一的 story/digest 内容模型，用于承接原始抓取字段、标准化后的统一字段、筛选与评分状态、栏目分类结果、页面展示字段和可选中文摘要字段。

原因：

- 避免采集、筛选、渲染各阶段各自维护不同结构。
- 便于测试 capability 边界。
- 便于对单条 story 的状态进行追踪和解释。
- 降低后续页面模板与数据处理的耦合。

### 4. 采用固定栏目编排，而不是每日自由结构

MVP 固定栏目为：今日头条、热门讨论、值得深读、Ask HN、Show HN。

原因：

- 页面结构稳定，读者预期明确。
- 规则可测试，便于 OpenSpec 编写。
- 有利于后续归档页统一展示。
- 降低日报编辑复杂度。

### 5. 首页直接作为最新日报，而不是独立门户页

首页不再是单独的信息架构层，而是“最新一期日报”的展示入口。

原因：

- 降低信息分叉与模板维护成本。
- 用户访问首页即可获得当天内容。
- 与“日报为核心产物”的产品定位一致。
- 历史列表可通过归档入口访问。

### 6. 历史归档采用静态可枚举结构

系统应保留历史日报归档，并使其可被静态页面访问。归档需要每期日报有稳定标识、历史期数不会被新构建破坏、用户能够从归档页浏览既往日报。

### 7. 标题始终保留原始标题，不做中文标题翻译

这是明确产品决策。原因：

- 避免翻译引入语义偏差。
- 降低摘要与翻译混用导致的信息噪声。
- 保持与 Hacker News 原内容的一致性。
- MVP 聚焦“精选与编排”，而非“重写标题”。

### 8. 中文摘要仅作可选增强，且只允许出现在特定栏目

中文摘要/推荐理由有严格边界：

- 仅适用于“今日头条”和“值得深读”。
- 每条最多一句。
- 可选出现，不是必填。
- 不影响原始标题展示。

### 9. 静态页面输出优先简单、可回退的结构

页面输出应优先选择简单稳定的静态结构，确保在 GitHub Pages 环境中无后端依赖，并且增强字段缺失时仍能产出基础页面。

### 10. 构建失败时优先局部降级，而不是整体中断

系统需要明确失败降级策略，优先级如下：

1. 保住历史归档不受影响。
2. 尽量生成当日日报，即使部分栏目不足。
3. 如果可选中文摘要缺失，仍发布纯英文标题版本。
4. 如果某些候选 stories 无法标准化，则跳过异常项。
5. 只有在无法形成最小可发布日报时，才将本次构建标记为失败。

## Risks / Trade-offs

- [离线生成牺牲实时性] → 接受日报是“日批次快照”，换取稳定性、可审计性和静态发布友好性。
- [GitHub Actions 受平台配额与定时精度影响] → 支持手动重跑，并确保失败不破坏已发布页面。
- [统一内容模型压缩特例表达能力] → MVP 优先围绕稳定的通用字段设计，后续再扩展特殊字段。
- [固定栏目降低编辑灵活性] → 配套栏目稀疏时的降级规则，但不为填满栏目而破坏分类准确性。
- [不翻译标题提高忠实性但提升中文用户门槛] → 通过头条/深读的一句中文说明做轻量补偿。
- [可选中文摘要带来内容质量差异] → 将其定义为可选增强，不作为生成日报的前置依赖。
- [静态归档长期增长] → MVP 先接受增量增长，后续如有必要再优化归档结构。
- [失败降级导致不同日期日报完整度不一致] → 以“稳定发布最小可用日报”优先于“等待完美内容后再发布”。
